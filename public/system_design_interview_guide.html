01
Product Design
"Design Twitter's News Feed"
The most common product design question. Tests your understanding of fan-out, caching, ranking, and the read/write asymmetry at social scale.
45-Minute Interview Script
0–3
MIN 0–3 · REQUIREMENTS
Clarify scope before drawing anything
"Before I start, let me clarify scope. Are we designing just the feed, or also tweet creation and following? Should I handle media — images and video? How many users — Twitter scale, around 300M DAU? Should the feed be real-time or near-real-time, say within a few seconds? I'll assume a Twitter-like product: 300M DAU, home timeline showing tweets from followed users, roughly 50 reads per user per day and 1 write per day."
3–8
MIN 3–8 · ESTIMATION
Back-of-envelope to justify your design
"300M DAU × 50 reads ÷ 86,400 = ~174K read QPS average, peak around 500K. Writes: 300M × 1 tweet ÷ 86,400 = ~3,500 write QPS. Read to write ratio is 50:1 — this is extremely read-heavy, which means we should aggressively cache feeds. Tweet storage: 300M tweets/day × 300 bytes = ~90 GB/day of text, manageable."
8–18
MIN 8–18 · HIGH-LEVEL DESIGN
Draw the core architecture
"Let me draw the high level. A user posts a tweet → hits the Write API → stored in Tweet DB (Cassandra, partition by user_id) → a Fan-out Service reads the author's follower list and pushes the tweet ID into each follower's pre-computed feed cache in Redis. On read, the Feed Service pulls from that Redis cache — instant response. This is fan-out on write."
18–30
MIN 18–30 · DEEP DIVE — THE HARD PART
Celebrity problem + hybrid fan-out
"The critical problem with pure fan-out on write: a celebrity with 50M followers posting one tweet would trigger 50M write operations synchronously — that's a thundering herd. My solution is a hybrid approach: for normal users, fan-out on write. For celebrities — defined as users with more than, say, 1M followers — we use fan-out on read. When a user loads their feed, we pull celebrity tweets separately and merge them with the pre-built feed. This keeps write costs manageable while reads stay fast."
30–40
MIN 30–40 · SCALE & BOTTLENECKS
Identify and solve weak points
"Three bottlenecks I want to address. First, the Redis feed cache — each user's feed stores ~200 tweet IDs, that's 200 × 8 bytes × 300M users = ~480 GB of cache. Totally feasible with a Redis cluster. Second, Cassandra for tweets — I'd shard by (user_id, tweet_id) to keep a user's tweets co-located. Third, the media — images go to S3, served through CloudFront CDN, not our origin servers."
40–45
MIN 40–45 · TRADE-OFFS & WRAP UP
Show maturity with honest trade-offs
"The hybrid fan-out adds complexity — we need to track who is a celebrity and update that threshold. Pre-computed feeds mean eventual consistency — a user might see tweets up to 2 seconds late, which is acceptable for a social network but wouldn't work for, say, a trading system. If I were building this, I'd also add a ML ranking layer between the feed cache and the response, to rank by engagement rather than pure chronological order."

Fan-out on write vs read vs hybrid. This IS the core problem.
②

50M follower user = 50M writes on one tweet. Interviewers always ask this.
③

Pre-compute timelines. 200 tweet IDs × 300M users = ~480 GB RAM.
④

Write-heavy, time-series, shards linearly. Partition by user_id.
⑤

Async fan-out. Decouple tweet write from feed propagation.
⑥

Senior Signal — Say These Things
  • Propose the hybrid approach unprompted
  • Mention eventual consistency trade-off explicitly
  • Discuss feed ranking beyond chronological order
  • Talk about cache eviction for inactive users
02
Infrastructure Design
"Design a Distributed Rate Limiter"
Tests your depth in algorithms, distributed coordination, and the trade-off between accuracy, speed, and consistency. A senior favourite.
45-Minute Interview Script
0–3
MIN 0–3 · REQUIREMENTS
Nail down what you're rate-limiting
"A few things I want to clarify: Are we rate-limiting per user, per IP, or per API key? What's the scale — 10K requests/sec? Should it be hard or soft limiting — block vs throttle? What accuracy do we need? A few percent error okay? I'll design a per-user rate limiter at 10K RPS, distributed across multiple API servers, with soft blocking — return 429 when exceeded. 1–2% error tolerance is acceptable."
3–12
MIN 3–12 · ALGORITHM CHOICE
Walk through all 4 algorithms — then pick one
"There are four common algorithms. Token Bucket — tokens refill at fixed rate, requests consume a token, allows bursting. Leaky Bucket — requests drain at fixed rate, no bursting. Fixed Window Counter — count reqs per minute, reset at window edge. Problem: 2× burst at window boundary. Sliding Window Log — track exact timestamps, most accurate but memory-heavy. Sliding Window Counter — approximate sliding window using weighted sum of current and previous window counts. For our use case I'd pick Sliding Window Counter — it's ~2% off from the exact log but uses O(1) memory per user and handles distributed counters cleanly in Redis."
12–25
MIN 12–25 · DISTRIBUTED DESIGN
The hard part — multiple nodes, shared state
"The distributed challenge: we have 10 API servers. If each stores its own count, a user can hit server A 100 times and server B 100 times and bypass a limit of 100. We need centralized counters. I'd use Redis with a Lua script — INCR and GET in a single atomic operation to avoid race conditions. The Redis key is `rate:{user_id}:{window}` with TTL = window size. For even higher scale, we shard Redis by user_id using consistent hashing — each user always maps to one Redis node, no cross-node coordination needed."
25–38
MIN 25–38 · DEEP DIVE
Redis failure, headers, and where to place it
"Three things I want to address proactively. One — Redis failure: if Redis goes down, we fail open by default, meaning requests go through without rate limiting. For a financial API, fail closed instead. Two — response headers: always return X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset so clients can back off intelligently. Three — placement: the rate limiter should sit in the API Gateway or as a middleware layer, not in each service — avoids duplicating the logic and keeps it auditable."
38–45
MIN 38–45 · TRADE-OFFS
Accuracy vs performance vs complexity
"The fundamental trade-off: more accurate = more memory and coordination. Token Bucket gives bursting which some APIs need. Sliding Window Log is perfect accuracy but O(n) memory. My sliding window counter gives 99% accuracy at O(1) memory. The final thing I'd add: a rules engine — don't hardcode limits in code. Store limit configs in a config service so you can update them without deploys."

Token bucket, leaky, fixed window, sliding window. Pick one and justify.
②
Race condition + atomic ops MUST
Redis Lua script for atomic INCR+GET. Multi-node race conditions.
③

Fail open vs fail closed — depends on system criticality.
④

X-RateLimit-Limit/Remaining/Reset. Shows API empathy.
⑤

Don't hardcode limits. Change rules without deploys.

  • Mention the fixed-window boundary burst problem
  • Discuss fail-open vs fail-closed explicitly
  • Propose a rules/config service for dynamic limits
  • Mention per-user vs per-IP vs per-endpoint limits
03
Data System Design
"Design YouTube's Analytics System"
Tests your understanding of stream vs batch processing, lambda/kappa architecture, OLAP vs OLTP, and data warehousing at scale.
45-Minute Interview Script
0–3
MIN 0–3 · REQUIREMENTS
Clarify the analytics queries first
"Before designing, I need to understand the queries. What metrics are we tracking — views, watch time, likes, revenue? What's the latency requirement — real-time dashboard, or is hourly okay? Who consumes this — creators, advertisers, internal teams? I'll design for: view counts and watch time, creators see near-real-time data with ~1 minute lag, supports historical queries going back 5 years, 500K videos with 1B views/day."
3–10
MIN 3–10 · ARCHITECTURE CHOICE
Lambda architecture — stream + batch
"I'll use Lambda Architecture. There are two layers: the Speed Layer handles near-real-time analytics — events flow through Kafka → Flink for stream processing → results in Redis for dashboards. The Batch Layer handles historical queries — the same events land in S3 data lake → Spark jobs run hourly/daily → results go into a data warehouse like Redshift. The Serving Layer merges real-time Redis counts with historical warehouse data to answer any query. The alternative is Kappa Architecture — stream only, no batch. Simpler but harder to correct historical data."
10–25
MIN 10–25 · DEEP DIVE — DATA FLOW
Walk through a single view event end to end
"A user watches a video. The video player fires a view event — JSON payload with user_id, video_id, timestamp, watch_duration, country — to an Event Ingestion API. This writes to Kafka topic `video_events`. From Kafka, two consumers: Flink aggregates views per video in tumbling 1-minute windows and updates Redis counters. Simultaneously, a Kafka consumer writes raw events to S3 in Parquet format partitioned by date. Hourly Spark jobs aggregate S3 data and load into Redshift. The creator dashboard queries the Serving Layer — recent data from Redis, older data from Redshift."
25–38
MIN 25–38 · DATA MODELING & SCALE
Schema design and deduplication
"Two important problems. First, deduplication: a user refreshing sends duplicate view events. I'd use a Bloom filter or a Redis SET with a 24-hour TTL storing (user_id + video_id + date) — if the key exists, discard the event. Second, the Redshift schema: I'd use a star schema — fact table `video_views` with foreign keys to dimension tables: dim_video, dim_user, dim_date, dim_geography. This enables fast GROUP BY queries like 'views by country for video X over last 30 days' with columnar storage."
38–45
MIN 38–45 · TRADE-OFFS
Lambda complexity vs Kappa simplicity
"Lambda's downside is maintaining two codepaths — one in Flink, one in Spark — that must produce consistent results. Teams often spend months debugging discrepancies. Kappa uses only streaming — Kafka with long retention acts as the historical store, you replay from the start to recompute. Simpler to maintain, but reprocessing 5 years of YouTube data takes days. My choice: Lambda for this because YouTube's historical query patterns are complex and batch SQL is far easier than streaming SQL for ad-hoc analytics."

②

Durable, replayable event log. Both speed and batch layer read from it.
③
OLAP vs OLTP distinction MUST
Analytics needs columnar (Redshift/BigQuery), not row-store (Postgres).
④

Fact + dimension tables. Enables fast aggregation queries.
⑤

Bloom filter or Redis SET to deduplicate view events.
⑥

Partition S3/Redshift by date for efficient time-range scans.

  • Discuss event deduplication strategy
  • Mention data partitioning for S3/Redshift
  • Bring up the Lambda codepath divergence risk

API Design
"Design a Payment Processing API"
Tests your understanding of idempotency, versioning, security, error contracts, and the nuance of designing APIs that are safe to retry and impossible to double-charge.
45-Minute Interview Script
0–5
MIN 0–5 · REQUIREMENTS
Clarify consumers, actions, constraints
"API design questions need very clear scope. Who calls this API — mobile apps, third-party merchants, our own services? What operations: initiate payment, check status, refund? Does it talk directly to card networks or wrap a PSP like Stripe? What are the consistency guarantees — can we double-charge a user? I'll design a merchant-facing REST API for charging customers: create charge, get charge status, issue refund. The non-negotiable: exactly-once semantics — a network timeout must never result in a double charge."
5–20
MIN 5–20 · API CONTRACT DESIGN
Design the actual endpoints with full detail
"Let me design the core endpoints. POST /v1/charges — body: {amount, currency, customer_id, payment_method_id, metadata}. Header: Idempotency-Key: {uuid}. Returns 201 with {charge_id, status: 'pending'|'succeeded'|'failed', amount, created_at}. GET /v1/charges/{charge_id} — returns full charge object with status. POST /v1/charges/{charge_id}/refunds — body: {amount (optional, for partial)}, returns refund object. The idempotency key is critical — if the same key is sent twice, return the cached response of the first request, never process twice. Store idempotency_key → response_id in Redis for 24 hours."
20–32
MIN 20–32 · DEEP DIVE — IDEMPOTENCY
The mechanism behind safe retries
"The idempotency flow: client sends POST /charges with Idempotency-Key: abc-123. Server checks Redis — key not found, process the charge. Store {key: 'abc-123', charge_id: 'ch_xyz', status: 202} in Redis with 24hr TTL. Return response. Client times out, retries with same key. Server checks Redis — key found, return cached response directly. No second charge. The tricky edge case: the first request succeeded but died before writing to Redis. We handle this with a database-level unique constraint on idempotency_key — so even if we reprocess, the DB insert fails and we return the existing charge."
32–40
MIN 32–40 · SECURITY & VERSIONING
Auth, PCI-DSS, and API evolution
"Security: all payment APIs use API keys in the Authorization header — never in the URL. Key format: `Authorization: Bearer sk_live_abc123`. Rate limit by API key. Never log raw card data — we're PCI-DSS scoped. Versioning: I prefer URL versioning (/v1/) over header versioning for discoverability. Maintain at least 2 versions simultaneously. Deprecate with a Sunset header 12 months ahead. When you break a contract, that's a major version bump. Additive changes — new optional fields — are non-breaking and can go in the same version."
40–45
MIN 40–45 · ERROR DESIGN & WEBHOOKS
Make errors as useful as the happy path
"Error contracts matter as much as success. Never return generic 500s. Return structured errors: {error: {code: 'card_declined', message: 'Your card was declined', decline_code: 'insufficient_funds', charge_id: 'ch_xyz'}}. HTTP 402 for payment failures, 422 for validation, 429 for rate limit. Webhooks: payments are async — the charge API returns 202 Accepted immediately. The actual result comes via webhook: POST to merchant's endpoint with {event: 'charge.succeeded', charge: {...}}. Include a webhook signature header so merchants can verify authenticity. Always retry webhooks with exponential backoff on 4xx/5xx from merchant."
Architecture Diagram — Payment API Flow Idempotency + Webhook Pattern
Must-Discuss Components
①
Idempotency keys MUST
Network retries must never cause double charges. Redis + DB unique constraint.
②
Structured error contracts MUST
Never return generic errors. Specific codes like `card_declined`, `insufficient_funds`.
③
Webhooks for async results MUST
Payments are async. Return 202, push result via signed webhook.
④
API versioning strategy MUST
/v1/ URL versioning. Sunset header. Additive changes are non-breaking.
⑤
Webhook signature security GOOD
HMAC-SHA256 signature so merchants verify event authenticity.
⑥
Partial refunds & states GOOD
Design clear state machine: pending → succeeded/failed → refunded.

Senior Signal — Say These Things
  • Explain idempotency mechanism end-to-end
  • Mention the Redis failure + DB unique constraint backup
  • Design error codes like a product, not just HTTP codes
  • Bring up webhook retry with exponential backoff
Universal Senior Engineer Checklist — Say These in Every Interview
  • Always state trade-offs explicitly — "I chose X over Y because... the downside is..."
  • Bring up observability unprompted — metrics, distributed tracing, alerting
  • Discuss failure modes — what happens when each component goes down?
  • Mention data consistency model — eventual vs strong consistency and why
  • Talk about rollout strategy — dark launches, feature flags, gradual rollout
  • Raise security considerations — auth, encryption, PII handling, audit logs
