The RESHADE Framework
STEP 01
Requirements Clarification
Never jump to design. Ask smart questions to narrow scope.
  • Functional requirements (core features)
  • Non-functional (scale, latency, availability)
  • Constraints (budget, tech stack)
  • Users & usage patterns
STEP 02
Estimation of Scale
Back-of-envelope math to justify design decisions.
  • DAU / MAU / QPS
  • Storage (5 yrs projection)
  • Bandwidth (read vs write ratio)
  • Memory / cache requirements
STEP 03
System Interface (API Design)
Define clear APIs before drawing boxes.
  • REST / GraphQL / gRPC choices
  • Key endpoints with params
  • Rate limiting strategy
  • Auth & versioning
STEP 04
High-Level Design
Sketch the big picture — clients, servers, DB, caches.
  • Core components diagram
  • Client ↔ Server ↔ DB flow
  • Data flow walkthrough
  • Monolith vs Microservices
STEP 05
Data Model & Storage
Choose storage thoughtfully based on access patterns.
  • SQL vs NoSQL decision
  • Schema / data model design
  • Indexing strategy
  • Sharding & replication
STEP 06
Algorithm & Core Logic
Dig into the tricky parts — the "how" of key features.
  • Feed ranking / recommendation
  • Search / typeahead logic
  • Unique ID generation
  • Consistent hashing
STEP 07
Deep Dive & Bottlenecks
Identify weak points and solve for scale.
  • Single points of failure
  • Hot spots / hot keys
  • Latency bottlenecks
  • Thundering herd problem
STEP 08
Evolution & Trade-offs
Show maturity by discussing what you'd change at 10x scale.
  • CAP theorem trade-offs
  • Cost vs performance
  • Observability / monitoring
  • Migration / rollout strategy

Do's & Don'ts
✓ DO
  • Always clarify requirements before drawing anything
  • Think out loud — interviewers want your reasoning
  • Start broad, then drill down on specific components
  • Explicitly state trade-offs for every decision
  • Use numbers to back up your choices
  • Ask "would you like me to go deeper here?"
  • Mention failure modes and how you'd handle them
  • Discuss observability — metrics, logs, alerts
✗ DON'T
  • Jump straight into the solution without scoping
  • Over-engineer for Day 1 (design for 10x, not 1000x)
  • Use buzzwords without explaining them
  • Ignore the non-functional requirements
  • Spend too long on one component — keep moving
  • Forget about security, auth, and edge cases
  • Be silent — narrate every thought
  • Say "it depends" without saying what it depends on
